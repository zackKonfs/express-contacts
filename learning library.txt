Step 3 — Create the .gitignore file
This must be done before npm install. If you run npm install first, node_modules gets created and you risk it being tracked by Git.

in terminal:
echo node_modules/ > .gitignore
echo .env >> .gitignore

Part             Meaning
echo             Print this text to somewhere
node_modules/    The text you want to write
>                Create a new file and write the text into it
.gitignore       The file to create and write into

So the whole line means: create a file called .gitignore and write node_modules/ into it.
If .gitignore already exists, > will overwrite it completely.

Part             Meaning
echo             Print this text to somewhere
.env             The text you want to write
>>               Open the existing file and ADD to the bottom
.gitignore       The file to append into

So the whole line means: open the existing .gitignore and add .env on a new line at the bottom.
The difference between > and >> is the key thing to remember:

> = overwrite — wipes the file and starts fresh
>> = append — adds to the bottom without touching what is already there

cat .gitignore

Part         Meaning
cat          Short for concatenate — but in practice it just means "show me the contents of this file"
.gitignore   The file you want to read

So cat .gitignore means: print the contents of .gitignore to the terminal so I can see what is inside it.
It is the quickest way to verify a file's contents without opening it in VS Code. You will use cat a lot when you want to quickly check what is inside a file.



npm init -y
npm init sets up your project as a Node.js project by creating a package.json file.



npm install express dotenv @supabase/supabase-js
npm install --save-dev nodemon

Part                      Meaning
npm install               Download these packages and save them to this project
express                   The web framework — handles routes, requests, responses
dotenv                    Loads your .env file variables into process.env
@supabase/supabase-js     The official Supabase JavaScript client — lets you talk to your database

All three get downloaded into node_modules and listed under "dependencies" in package.json. These are packages your app needs to actually run — on your machine AND on Render.

Part              Meaning
npm install       Download this package
--save-dev        Mark it as a development-only tool
nodemon           Watches your files and automatically restarts the server when you save

nodemon gets listed under "devDependencies" in package.json instead of "dependencies". Render ignores devDependencies when it deploys — it only installs what your app actually needs to run, not your local developer tools.


Open your project in VS Code:
code .


add below to package.json:
  "dev": "nodemon server.js"

Right now if you want to start your server you would have to type:
bashnodemon server.js
Every single time. That is fine but it means you need to remember the exact command and the exact filename.
When you add this line to package.json:
json"dev": "nodemon server.js"
You are creating a shortcut command. Now instead of typing nodemon server.js you just type:
bashnpm run dev
And npm looks up what dev means in your scripts, finds nodemon server.js, and runs it for you.

Line 1:
git remote add origin https://github.com/YOUR_USERNAME/express-contacts.git

Part                      Meaning
git remote add            Add a remote connection to this local project
origin                    The nickname for that remote connection — "origin" is the universal convention for your main GitHub repo
https://github.com/...    The actual URL of your GitHub repo

This line is saying: "Hey Git, I want you to know about a remote location called 'origin', and it lives at this GitHub URL."
Before this line, your local folder and GitHub know nothing about each other. After this line, Git knows where to send your code when you push.

Line 2:
git remote -v
Part                     Meaning
git remote               Show me the remote connections for this project
-v                       Verbose — show the full URLs, not just the names

This is just a verification step. It prints out the remote connections so you can confirm the URL is correct before you try to push anything.
You should see something like:
origin  https://github.com/zackx/express-contacts.git (fetch)
origin  https://github.com/zackx/express-contacts.git (push)
Two lines — same URL — one for fetching (downloading) and one for pushing (uploading). That is normal and correct.


if wrong, to undo:
git remote remove origin


extra:
git rm -r --cached .
Part               Meaning
git rm             Tell Git to remove files from tracking
-r                 Recursive — do this for all files inside all subfolders too
--cached           Remove from Git's tracking only — do NOT delete the actual files from your hard drive
.                  Everything in the current folder



extra (only need once. its about Mac use LF and window use CRLF and convert all LF to CRLF in window):
git config --global core.autocrlf true
Part              Meaning
git config        Change a Git setting
--global          Apply this setting to ALL projects on your machine — not just this one
core.autocrlf     The setting that controls how Git handles line endings
true              On Windows, automatically convert line endings when committing
